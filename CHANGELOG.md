# CHANGELOG — BuhloWar

## 2026-03-31 / 2026-04-01

### index.html / index300326.html — Веб-интерфейс

- **Уведомления TSA:** Добавлена проверка температуры TSA при достижении лимита. При срабатывании — звуковой сигнал (Web Audio API) + браузерное уведомление (Notification API). Однократное срабатывание (флаг `prevTsaAlarmState`), повторное — только после снижения ниже лимита.
- **Блок ТЕЛО (МОНИТОРИНГ):** Объединены ПРОГНОЗ и ОТОБРАНО в единый формат `--/-- мл` по аналогии с блоком ГОЛОВЫ. Удалены дублирующие строки.
- **Мастер калибровки клапанов:**
  - Исправлены URL-вызовы — заменён захардкоженный `/api/command` на универсальные `getCloudCommandUrl()` (работает через облако и локально с SD-карты).
  - Автоматическое закрытие окна мастера через 2 сек после расчёта пропускной способности.
  - Ввод объёма: целые числа (шаг 1 мл вместо 0.1). При переходе на шаг ввода значение инициализируется из текущего `cap` в настройках (оператор редактирует, а не вводит с нуля). Защита от сброса при обновлении статуса.

### ProcessEngine.cpp — Движок процессов

- **NEXT_STAGE после калибровки:** Исправлен баг — блок `if (currentStage == VALVE_CAL) { return OK; }` стоял ДО обработчика `NEXT_STAGE` и глотал команду кнопки ДАЛЕЕ. Обработчик `NEXT_STAGE` перенесён перед блоком `VALVE_CAL`.
- **Команды калибровки из Web:** Исправлен баг — команды `CALIB_START_DRY`, `CALIB_START_CAPACITY`, `CALIB_CANCEL` перехватывались ранним `return` в блоке `VALVE_CAL`. Перенесены ДО проверки этапа.
- **Отладка NEXT_STAGE:** Добавлен лог `[WebCmd] NEXT_STAGE received, currentStage=X` для диагностики переходов.
- **Пересчёт оставшегося времени ТЕЛО (Шпора):** Старая формула `(predictVol / speedShpora) - elapsedSec` завышала время при снижении скорости, т.к. предполагала что текущая (сниженная) скорость была с самого начала. Новая формула `(predictVol - bodyVolDone) / speedShpora` правильно считает остаток объёма при текущей скорости.

### CloudManager.h — Облачный менеджер

- **Kernel Panic fix:** `WiFiClientSecure` теперь поле класса (`_client`), а не локальная переменная в каждом методе. Ранее каждый вызов `sendTelemetry()`/`checkCommands()`/`checkSettings()` создавал новый TLS контекст (~20KB), вызывая фрагментацию кучи. Инициализация через `ensureClient()` — один раз при первом вызове.

### AppNetwork.h / AppNetwork.cpp — Сетевой менеджер

- **Удалена зависимость UniversalTelegramBot:** `#include <UniversalTelegramBot.h>` закомментирован. Убраны: создание `bot`, загрузка `TELEGRAM_CERTIFICATE_ROOT` (~1.5KB RAM), очередь сообщений Telegram. Экономия ~20KB RAM.
- **Очистка:** Удалены реализации методов `queueMessage`, `isTelegramReady`, `sendTelegramNow`, `processMessageQueue` (были без объявлений в .h после чистки Telegram).
