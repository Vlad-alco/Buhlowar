### ProcessEngine.cpp + index.html — WEB мониторинг тела в Стандарт методе: отображение при NASEBYA и фикс фантомного объёма
`cc40528`

**Проблема 1: WEB не показывает данные тела при переходе "на себя"**
- При залёте в Стандарт методе система переходит в NASEBYA (стабилизация "на себя"). WEB мониторинг показывал `"--"` для метода, скорости, цикла, расч/факт.
- C++ `updateRectWebInfo()` обнулял `bodyMethodName`, `bodySpeed`, `bodyCycle` при любом этапе кроме TELO.
- JS `index.html` показывал блок тела только при `stage == 'TELO'`.
- **Фикс C++**: добавлена ветка `else if (currentStage == Stage::NASEBYA)` — при "на себя" данные тела сохраняются, не обнуляются.
- **Фикс JS**: три места в `index.html` изменены с `stage == 'TELO'` на `stage == 'TELO' || stage == 'NASEBYA'`. При NASEBYA рядом с циклом показывается пометка `⏳`.

**Проблема 2: Фантомный объём при возврате из NASEBYA**
- При возврате из NASEBYA в TELO первый расчёт `dt_h` охватывал всё время "на себя" (10-20 мин), и `bodyVolDone += скорость × время_NASEBYA` прибавлял объём за время когда клапан был закрыт.
- **Фикс**: `lastVolCalcTime = millis()` перед `changeStage(Stage::TELO)` в `handleNasebya()`. Отсчёт интервала начинается заново, `bodyVolDone` сохраняется и продолжает корректно накапливаться.

**Влияние на Шпору**: Шпора при залёте вызывает `finishTelo()` и никогда не переходит в NASEBYA. Ни одно изменение не затрагивает логику Шпоры.

### CloudManager.h — FreeRTOS задача: нулевая блокировка loop
`0ccf4ae`

- **Все HTTP-запросы вынесены в отдельную FreeRTOS задачу на core 0** (там WiFi/TCP стек). Основной loop на core 1 больше никогда не блокируется, независимо от состояния сервера.
- **Архитектура**: cloud задача (core 0, приоритет 0, стек 8KB) выполняет все HTTP-запросы, backoff, управление `_client`. Основной loop вызывает `update()` — который копирует телеметрию через mutex (~2 мкс), читает очереди команд/настроек, вызывает callbacks. Задержка — ноль.
- **Связь loop↔cloud**: телеметрия — mutex (запись из loop, чтение из cloud); команды и настройки — FreeRTOS очереди (cloud→loop, неблокирующее чтение).
- **Callbacks** `onCommand`/`onSettings` вызываются из контекста основного loop — безопасно, без race conditions.
- **Ресурсы**: ~16KB heap (8KB стек задачи + 7KB очереди + 1KB overhead).
- Публичный API идентичен — **никаких изменений в других файлах**.

### CloudManager.h — _client.stop() только при ошибках
`8d9caea`

- Предыдущая версия вызывала `_client.stop()` перед каждым запросом, что заставляло делать новый TLS handshake каждый раз (2-5 сек overhead). Web-интерфейс с SD карты начал притормаживать, система зависала.
- Теперь `_client.stop()` вызывается **только при `errorCount > 0`** (предыдущий запрос завершился ошибкой = мёртвое TCP соединение). При успехе соединение переиспользуется с нулевой задержкой.

### CloudManager.h — Предотвращение 3-минутного зависания loop
`c302901`

- **Причина**: при недоступности сервера TCP ретрансмиссии на переиспользуемом мёртвом соединении блокировали loop на 1→2→4→8→16 сек суммарно до 3 минут.
- **Решение**: `_client.stop()` перед `http.begin()` (закрывает мёртвое TCP), экспоненциальный backoff при ошибках (интервалы: 2→4→8→16→32 сек, макс. множитель x5), обновление lastCheckMs даже при ошибке.
- Добавлено логирование backoff: `[Cloud] Backoff: errorCount=N, next interval=X ms`.

### CloudManager.h + api.php — Fix ложного errorCount от settings check
`dabe4f6`

### Анализ: LOOP Slow 56-126 мс — не является проблемой
Исследование, без изменений кода

- `[LOOP] Slow: 56-126 ms` появляется ~2.5 раза/сек — это нормально для контроллера.
- Причина: `buildTelemetryJson()` вызывается каждый loop (~150 раз/сек), делает 60+ конкатенаций String + `logger.readNewLog()` открывает SD-файл. На слабых итерациях совпадает с обновлением DS18B20/BME280.
- 56-126 мс не влияет на работу: процессы идут с секундными интервалами, LCD раз в 500 мс.
- NetMgr, CloudManager и MEM check — выполняются в отдельных FreeRTOS задачах на core 0, не блокируют loop на core 1.
- Если потребуется ускорить — вызывать `buildTelemetryJson()` раз в 2 сек вместо каждого loop.

### AppNetwork.h/cpp + BuhloWar.ino — Fix Kernel Panic при загрузке от раннего HTTP запроса
`5041d94`

- **Причина**: NetworkTask запускается на core 0 до `processEngine.begin()` (строка 188 vs 271 в setup). Если браузер успевал сделать запрос в это окно (~1-2 сек), `processEngine->getSensorData()` обращался к `sensorAdapter = nullptr` → Core 0 panic (LoadProhibited).
- **Решение**: добавлен флаг `systemReady` в AppNetwork. Все вызовы `server->handleClient()` защищены проверкой `systemReady`. API handlers недоступны до полной инициализации системы.
- `appNetwork.setSystemReady(true)` вызывается в setup() после **всей** инициализации (строка 292) — после `processEngine.begin()`, создания всех меню и LCD.
- Защищены все точки вызова `handleClient()`: `update()`, `beginNetwork()`, `connectToWiFi()` (обе сети), `checkInternet()`.

### Документация: REF TSAR/REF АД и коррекция давления в TELO
Вопросы при тестировании Стандарт метода:

- **REF TSAR** (`rtsarM`) и **REF АД** (`adPressM`) в WEB мониторинге — это расчётные значения, зафиксированные **один раз** при входе в TELO (`rtsarM < 0.1`). Не меняются весь процесс, ни при залётах, ни при переходах NASEBYA → TELO. Обнуляются только при сбросе/начале новой колонны. Одинаковы для Стандарт и Шпора.
- **Коррекция давления** `(data.pressure - adPressM) * 0.0278` — применяется в обоих методах:
  - **Шпора**: учитывается для delta (снижение скорости) и hysteresis (залёт).
  - **Стандарт**: учитывается только для hysteresis (залёт → NASEBYA или finishTelo). Delta нет, так как в Стандарте нет механизма плавного снижения скорости — только бинарное решение "залёт / не залёт".
  - При росте давления (фронт с низким уходит): пороги повышаются → залёт позже. При падении: пороги снижаются → залёт раньше.
